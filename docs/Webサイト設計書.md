# 歎異抄・教行信証 読解 Webサイト 設計書

## 1. 文書の位置づけ

本書は、`doc/` 配下のテキスト文献を一次資料とし、閲覧・検索・現代語訳を提供する Web アプリケーションの**機能・情報・データ・API・運用**を定義する。実装の詳細（個別コンポーネント名など）は本書の範囲外とし、フェーズごとに別紙または Issue で補足する。

---

## 2. 目的と非目的

### 2.1 目的

- **信頼できる本文**を、書・大章・条（または教行信証の区分）単位で一貫した URL とともに提示する。
- **教行信証の全体像**を『教行信証大意』で示し、『顕浄土真実教行証文類』本文へ誘導する。
- **現代語訳**は、**スクリプトによりあらかじめ AI が翻訳した成果物**を取り込み、サイト上では原文とあわせて**表示するのみ**とする（利用者操作でその場 LLM を呼ばない）。
- 将来的に **用語・テーマの横断**（歎異抄と教行信証の関連）をデータとして蓄積できる土台を用意する。

### 2.2 非目的（初期リリースで行わない）

- 学術出版レベルの校訂・註釈書の代替。
- 利用者操作に応じた**その場での** LLM 現代語訳（初期リリースでは行わない。バッチ成果物の閲覧に限定する）。
- API キーをブラウザに配布するクライアント直叩き（セキュリティ上禁止）。

---

## 3. 環境変数（現代語訳バッチ・LLM）

プロジェクトルートの `.env` に以下を定義し、**現代語訳バッチ（CLI スクリプト、例: `npm run translate`）実行時**にのみ参照する。訳文をリポジトリまたはビルド成果物に取り込む**静的配信のみ**の本番では、ホスティングに API キーを置かなくてよい。

| 変数名 | 用途 |
|--------|------|
| `ANTHROPIC_API_KEY` | Anthropic API 認証 |
| `ANTHROPIC_MODEL` | Anthropic で使用するモデル識別子 |
| `OPENAI_API_KEY` | OpenAI API 認証 |
| `OPENAI_MODEL` | OpenAI で使用するモデル識別子 |

### 3.1 運用上の注意

- `.env` は **Git に含めない**（`.gitignore` で除外済みを前提とする）。
- **CI や管理用ランナー**でバッチを実行する場合のみ、ホスティング／CI のシークレットに API キーを注入する。訳文をコミットし**静的配信のみ**とする本番オリジンには、LLM 用キーを置かなくてよい。
- **キーがリポジトリやログに混入した場合は直ちにローテーション**する。

### 3.2 プロバイダの使い分け（推奨ポリシー）

設計上は両方を利用可能とするが、次のいずれかを実装時に固定することを推奨する。

| 方針 | 内容 |
|------|------|
| A. 単一プロバイダ | バッチ現代語訳は `ANTHROPIC_*` または `OPENAI_*` の一方のみ使用。他方は切替・検証用に予約。 |
| B. 役割分担 | 例：初訳を一方、推敲・用語統一を他方（オフラインの二段パイプラインとして運用）。 |

**初期実装**は方針 A（単一）でよい。切替はスクリプトの `--provider` または環境変数 `TRANSLATION_PROVIDER=anthropic|openai` で行えると運用しやすい（本書では任意機能として記載）。

## 4. 情報アーキテクチャ（サイトマップ）

```
/                          トップ（二書の案内、読み方）
/teachings/kyogyo/         教行信証ハブ
  /overview/               教行信証大意（全文・整形）
  /text/                   教行信証本文インデックス
  /text/[sectionId]/       各区分の本文（sectionId は下表）
/tannisho/                 歎異抄ハブ
  /preface/                序
  /section/[id]/           各条
  /appendix/ryuzaikiroku/  流罪記録
/glossary/                 用語索引（フェーズ2以降）
/search                    検索（フェーズ2以降・全文またはインデックス）
/about                     底本・データ由来・免責
```

### 4.1 教行信証本文の `sectionId`（一次データの `#2` に対応）

| sectionId（例） | 内容 |
|-----------------|------|
| `sosho` | 総序 |
| `kyo-bunrui-1` | 教文類一 |
| `gyo-bunrui-2` | 行文類二 |
| `shin-bunrui-3-hon` | 信文類三（本） |
| `shin-bunrui-3-matsu` | 信文類三（末） |
| `sho-bunrui-4` | 証文類四 |
| `shin-butsudo-bunrui-5` | 真仏土文類五 |
| `keshin-butsudo-bunrui-6-hon` | 化身土文類六（本） |
| `keshin-butsudo-bunrui-6-matsu` | 化身土文類六（末） |

実装ではマッピングテーブルをコードまたはデータで一元管理する。

### 4.2 歎異抄の URL

- 序：`/tannisho/preface`
- 第 n 条：`/tannisho/section/[n]`（内部 id は連番でも全角数字マッピングでもよいが、URL は ASCII 推奨）
- 流罪記録：`/tannisho/appendix/ryuzaikiroku`

## 5. データ設計

### 5.1 ソースファイルと前処理

| ファイル | エンコーディング | 処理 |
|-----------|------------------|------|
| `doc/19 歎異抄Ｇ.txt` | UTF-8 | `#1` `#2` `P--` 等をパース |
| `doc/04 教行証Ｇ.txt` | Shift_JIS | UTF-8 へ変換後、同一ルールでパース |
| `doc/23 教行意Ｇ.txt` | UTF-8 | 章立てが単純なら Markdown 化も可 |

`04` に含まれる `&M003254;` 等のエスケープは、パース仕様で「欠字・特殊記号・外部参照」のいずれかに分類し、表示層でルール化する。

### 5.2 正規化後の論理スキーマ（最小）

```text
Work
  id, title, slug

Section
  id, work_id, parent_id, order, title, source_marker  // source_marker: 元 txt の #2 等

Paragraph
  id, section_id, order, body_plain, annotations[]     // 括注を構造化できる場合は分離

TranslationArtifact（現代語訳・事前生成）
  paragraph_id または section_id への参照
  provider, model, locale  // 例: anthropic, ANTHROPIC_MODEL の値, ja-mod
  body, generated_at, prompt_version  // バッチ実行日時・プロンプト版（原文改訂時の差し替え判断用）
  source_path  // 任意: 生成元となった中間ファイル（例: content/translations/...）
```

永続先は初期は **リポジトリ内の JSON／Markdown チャンク**（`content/translations/` 等）または **SQLite**。トラフィック増に応じて RDB に移行可能とする。原文パース結果と訳は**ビルド時に結合**し、サイトは静的に配信する。

## 6. 現代語訳機能の設計

### 6.1 基本方針

- 現代語訳は **CLI スクリプト（例: `scripts/modern-translate.mjs`）** を開発者または CI が実行し、**あらかじめ AI で翻訳したテキスト**を成果物として保存する。
- Web アプリは **保存済みの訳を読み込み表示するのみ**とし、閲覧者の操作で LLM API を呼ばない。
- LLM の API キーは **バッチ実行環境**（ローカル、CI、管理用ランナー）にのみ配置する。静的ホストのみの本番ではキー不要。

### 6.2 バッチの入力単位（スクリプト側）

| 方式 | 利点 | 欠点 |
|------|------|------|
| 段落／行バッファ単位 | 再実行時の差分が小さく、失敗の巻き戻しがしやすい | API 回数が増えやすい |
| 節・条単位 | 回数削減・文脈一貫性 | トークン上限・1 回の失敗コスト |

**推奨**：既存スクリプトに合わせ、行バッファ＋最大文字数によるチャンク化を基本とする。取り込みパイプライン側で `Paragraph` 単位へ正規化してもよい。

### 6.3 プロンプト設計（要件）

バッチのシステム／ユーザプロンプトに必ず含める要素：

- 役割：宗教学の代わりではなく、**読みやすい現代日本語**への翻案。
- 敬称・仏教用語：過度な口語化を避け、**用語の初出では必要に応じて括弧で原語または漢字表記**を残す方針を明示。
- **出典の追加禁止**（ハルシネーション抑制）：文献にない事実を補わない。
- 入力が古典・漢文まじりの場合は、**意味の推定に迷う箇所は訳注として短く明示**してよい、とするか否かを方針として固定する。

### 6.4 成果物の更新と版管理

- 訳ファイルの先頭またはメタデータに **使用モデル・生成日時・プロンプトバージョン**を記録する（スクリプトが既に付与する形式を流用してよい）。
- 原文（`doc/*.txt`）またはプロンプトが更新されたら、**該当チャンクのみバッチを再実行**し、`TranslationArtifact` を差し替える。
- 任意：Git で訳文をバージョン管理し、Pull Request で人間によるレビューを挟む。

### 6.5 コスト・運用

- コストは **バッチ実行回数**に集約される。本番トラフィックに比例しない。
- 長大 `section` はスクリプト側でチャンク分割し、接続不安定時はリトライ・タイムアウト・キャッシュディレクトリ等の運用パラメータで調整する（既存スクリプトのオプションに準拠）。

### 6.6 表示 UI

- デフォルト：**原文のみ**。
- トグルまたはタブ：**原文 | 現代語訳 | 並列（左右／上下）**。
- 訳の下に小さく **使用モデル名および生成日時**（成果物に記録された値）を表示する（透明性）。

## 7. フロントエンド設計（概要）

### 7.1 技術スタック（推奨）

- **Next.js（App Router）** + TypeScript：静的ページとサーバ API の同居が容易。
- スタイル：**CSS Modules または Tailwind**（プロジェクト方針で確定）。
- フォント：本文は明朝系、UI はゴシック系の組み合わせを推奨。

### 7.2 主要画面要素

- **サイドナビ**：現在の `Work` / `Section` のツリー。
- **本文エリア**：`Paragraph` のループ、括注の視覚区別。
- **教行信証**：縦書きは任意。初期は横書きで可読性優先、CSS `writing-mode` で後から対応可能にする。

### 7.3 アクセシビリティ

- 見出し階層（h1→h2→h3）を URL 構造に対応させる。
- キーボードで目次・本文にフォーカス移動可能にする。

## 8. バックエンド・ビルドパイプライン

1. **取り込み**：`scripts/ingest.mjs` で `doc/*.txt` を読み、UTF-8 統一・パース、`content/corpus.json` に書き出す。
2. **現代語訳バッチ**：`npm run translate` 等で訳文を生成し、`content/translations/` 等へ保存する（手動または CI）。
3. **静的生成**：可能な範囲で `Section` ごとに SSG。原文と事前生成訳をビルド時に結合してページ化する。
4. **本番**：静的ホストの場合は鍵不要。ビルド時のみ鍵が要る構成なら、ビルド用シークレットを注入する。

---

## 9. セキュリティ

- API キーは **バッチ・ビルド環境のみ**。クライアントバンドルに `process.env.*` を漏らさない（Next.js の `NEXT_PUBLIC_` プレフィックスを付けない）。
- 訳文を静的配信する場合、**ランタイムに LLM キーを置かない**運用を既定とする。
- ログにリクエストボディ全文を残す場合は、個人情報・大量テキストの保管方針を決める。
- CORS：自サイトオリジンのみ（API を別ドメインにする場合）。
